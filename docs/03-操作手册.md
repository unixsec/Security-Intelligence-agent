## 一、日常运维操作

### 1.1 查看系统状态

```bash
# 查看所有 Pod 运行状态
kubectl get pods -n intel-system

# 查看 CronJob 执行历史
kubectl get jobs -n intel-system
kubectl describe cronjob collector-rss -n intel-system

# 查看最近日志
kubectl logs -n intel-system -l app=intel-admin-api --tail=100
kubectl logs -n intel-system -l app=collector-rss --tail=100
```

### 1.2 手动触发采集

```bash
# 手动触发 RSS 采集（创建一次性 Job）
kubectl create job --from=cronjob/collector-rss \
  manual-rss-$(date +%s) -n intel-system

# 手动触发搜索采集
kubectl create job --from=cronjob/collector-search \
  manual-search-$(date +%s) -n intel-system
```

### 1.3 手动触发报告生成

```bash
# 通过 Dify API 手动触发日报
DIFY_KEY=$(kubectl get secret dify-api-credentials -n intel-system \
  -o jsonpath='{.data.api-key}' | base64 -d)

curl -X POST \
  http://dify-api.intel-system.svc:5001/v1/workflows/run \
  -H "Authorization: Bearer ${DIFY_KEY}" \
  -H "Content-Type: application/json" \
  -d '{
    "inputs": {"report_type": "daily", "report_date": "'"$(date +%Y-%m-%d)"'"},
    "user": "manual-trigger"
  }'
```

## 二、情报源管理

### 2.1 新增 RSS 情报源

1. 访问管理控制台 → 情报源管理
2. 点击「新增情报源」
3. 填写：
   - 类型：RSS
   - 名称：例如 "HackerNews Security"
   - URL：RSS Feed 地址
   - 分类：选择对应分类
   - 优先级：high/medium/low
4. 保存后，在下一轮 CronJob 执行时生效

### 2.2 批量导入情报源

CSV 格式示例（UTF-8 编码，有表头）：

```csv
source_type,name,url,category,language,priority,notes
rss,FreeBuf,https://www.freebuf.com/feed,general_security,zh,high,
rss,Hacker News,https://hnrss.org/newest?points=100,general_security,en,medium,
website,CISA Advisories,https://www.cisa.gov/news-events/advisories,general_security,en,high,playwright
```

> 备注字段含 "playwright" 表示该站点需要 JS 渲染，将使用 Playwright 抓取。

### 2.3 评估情报源质量

通过健康分和采集量判断情报源价值：

| 健康分 | 状态 | 处理建议 |
|--------|------|---------|
| 100 | 正常 | 无需处理 |
| 70-99 | 有偶发失败 | 观察 |
| 50-69 | 频繁失败 | 检查 URL 是否变更 |
| 0-49 | 严重异常 | 暂停或修复 |

## 三、评分模型管理

### 3.1 查看当前评分配置

访问管理控制台 → 系统配置 → 评分模型配置

### 3.2 调整评分权重（热更新）

在管理控制台「系统配置」中直接调整权重后保存，配置即时生效（无需重启）。

推荐权重调整原则：
- 威胁可能性（0.25）：情报是否有实际利用价值的核心指标
- 业务影响度（0.30）：对本企业业务影响的最重要维度
- 合规影响度（0.20）：法规/监管相关情报的关键维度
- 时效紧迫性（0.15）：控制情报的响应优先级
- 情报质量（0.10）：来源可信度的基础过滤

### 3.3 使用脚本升级评分模型

```bash
# 1. 准备新模型 JSON 文件（参考 database/seed.sql 中的 scoring_model 格式）

# 2. 执行升级（先回测，再确认）
python scripts/migrate-scoring-model.py \
  --db-host localhost \
  --db-user intel_admin \
  --db-password 'your_password' \
  --new-model-file new_model_v2.json \
  --backtest-size 100

# 3. 验证升级结果
# 在管理控制台系统配置页查看版本号
```

## 四、监控与告警

### 4.1 访问 Grafana 监控看板

访问：`http://grafana.internal.company.com`

主要仪表盘：
- **Intel Agent Overview**：系统整体运行状态

| 指标 | 正常值 | 告警阈值 |

| 日采集量 | 50-200 条 | < 前7日均值50% |

## 五、LLM 配置管理

### 5.1 切换 LLM 模型

在管理控制台「系统配置」→ LLM 模型配置中修改：

```json
{
  "provider": "qwen",
  "api_endpoint": "http://qwen-server.llm-serving.svc.cluster.local:8000/v1",
  "model_name": "qwen2-72b-instruct",
  "timeout": 60,
  "max_retries": 3,
  "max_concurrency": 10
}
```

保存后立即生效（热更新）。

### 5.2 验证新 LLM 连通性

```bash
python scripts/test-llm.py \
  --api-url http://qwen-server.llm-serving.svc:8000/v1 \
  --api-key your-key \
  --model qwen2-72b-instruct
```

## 六、Dify Workflow 运维

### 6.1 查看 Workflow 执行状态

登录 Dify 控制台 → 「监控」页面可查看：
- 各 Workflow 的执行历史
- 失败的执行记录和错误原因
- LLM 调用日志

### 6.2 Workflow 执行失败处理

若 WF-2 分析失败：
1. 查看 Dify 日志中的错误原因
2. 检查 raw_intel 表中对应记录的 status 字段
3. 若为 LLM 超时，检查 LLM 服务资源利用率
4. 手动重试：在 Dify 控制台找到失败的执行记录 → 重试

### 6.3 更新 Workflow（Prompt 调优）

1. 在 Dify 控制台编辑对应 Workflow
2. 修改 LLM 节点的 System Prompt 或 User Prompt
3. 使用「测试」功能验证效果
4. 发布新版本

## 七、数据库维护

### 7.1 数据库备份

```bash
# 进入 MySQL Pod
kubectl exec -it -n intel-system \
  $(kubectl get pods -n intel-system -l app=mysql-primary -o jsonpath='{.items[0].metadata.name}') \
  -- bash

# 执行备份
mysqldump -u root -p intel_system > /tmp/intel_backup_$(date +%Y%m%d).sql

# 从 Pod 复制备份文件到本地
kubectl cp intel-system/<mysql-pod-name>:/tmp/intel_backup_YYYYMMDD.sql \
  ./backups/intel_backup_$(date +%Y%m%d).sql
```

### 7.2 清理过期数据

```sql
-- 清理 30 天前的原始情报（已分析且重复的）
DELETE FROM raw_intel 
WHERE status IN ('analyzed', 'duplicate') 
  AND created_at < DATE_SUB(NOW(), INTERVAL 30 DAY);

-- 清理 90 天前的推送日志
DELETE FROM push_logs WHERE pushed_at < DATE_SUB(NOW(), INTERVAL 90 DAY);

-- 清理 180 天前的审计日志
DELETE FROM audit_logs WHERE created_at < DATE_SUB(NOW(), INTERVAL 180 DAY);
```

## 八、故障处理快速参考

| 现象 | 可能原因 | 处理方法 |
|------|---------|---------|
| 日报未收到 | CronJob 未执行 / Dify Workflow 失败 | 检查 CronJob 状态和 Dify 日志 |
| 情报采集量为 0 | 情报源全部失败 / RSS URL 失效 | 检查情报源健康状态 |
| LLM 评分结果异常 | Prompt 效果差 / LLM 服务不稳定 | 在 Dify 中测试 WF-2，调整 Prompt |
| P0 告警未推送 | 企业微信 Webhook 失效 | 重新获取 Webhook URL 并更新 Secret |
| MySQL 连接超时 | 连接池耗尽 / MySQL 负载过高 | 增加连接池大小或优化慢查询 |
| Milvus 去重失效 | Milvus 服务重启后集合未加载 | 执行 `collection.load()` 重新加载 |
